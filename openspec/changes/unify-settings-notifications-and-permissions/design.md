## Context

目前專案已經完成留言、通知鈴鐺、團隊看板與標準工具列，但設定頁仍保留舊版區塊式樣，通知中心也只有單筆已讀，權限控制仍以 `Admin / Editor / Leader` 這種硬編碼判斷為主。參考系統的權限管理頁面顯示，實務上需要拆成「使用者角色管理」與「角色權限設定」，並且要能同時控管頁面存取與功能操作。

這次變更會同時碰到 UI 一致化、通知互動、資料模型與權限決策，是跨多個模組的變更，需要先把方向寫清楚，避免後面實作時每頁各自長不同邏輯。

## Goals / Non-Goals

**Goals:**
- 讓設定頁的工具列、篩選列與操作區視覺風格和其他主頁一致。
- 讓通知中心支援「全部標記已讀」與一致的未讀摘要更新。
- 導入角色權限模型，讓頁面顯示與操作按鈕不再只靠固定角色常數判斷。
- 參考 ContractHub 的結構，定義角色、可存取頁面與可執行動作的關係。
- 保留目前團隊 / Leader 的既有資料，讓新權限模型可以漸進接入。

**Non-Goals:**
- 不在這次引入完整後端權限審計或操作紀錄。
- 不處理瀏覽器推播、通知分類器或通知偏好設定。
- 不在這次完成 NAS 部署或跨專案權限共享。
- 不重做整個設定頁資訊架構，只統一風格並加入權限管理能力。

## Decisions

### 1. 設定頁視覺統一沿用既有 `StandardToolbar`

- 決定：不再為設定頁另外設計一套表單工具列，直接延用目前主頁已落地的 `StandardToolbar` 與同樣的圓角、邊框、間距規則。
- 原因：現在這套風格已經在待辦、問題、會議、團隊看板驗證過，擴充成本最低，也能避免之後再出現第二套樣式。
- 替代方案：
  - 另外為設定頁做獨立樣式：彈性較高，但會重新分裂視覺系統。
  - 整頁重做：成本太大，且和本次需求不成比例。

### 2. 通知「全部已讀」採前端批次更新 + 摘要重算

- 決定：通知中心新增「全部標記已讀」按鈕，前端對目前使用者的未讀通知做批次更新，更新後同步重算未讀 badge。
- 原因：目前通知資料已經是以每個使用者自己的集合管理，這種模式最容易往前擴充。
- 替代方案：
  - 僅做 UI 假動作：會造成 badge 與實際資料不同步。
  - 改成集中式通知摘要文件：之後可做，但這次沒有必要先重構。

### 3. 權限模型拆成「角色定義」與「使用者指派」

- 決定：新增角色權限定義資料，將頁面存取與功能操作拆分；使用者則只記錄目前被指派的角色。
- 原因：這種模式和參考系統一致，也比較容易在不同頁面統一判斷。
- 替代方案：
  - 繼續直接在 user 文件上存多個布林值：短期能用，但長期很難維護。
  - 僅保留 Admin / Editor / Leader / Member 固定判斷：不夠細，也不符合使用者需求。

### 4. 保留舊角色邏輯作為相容層

- 決定：新權限系統接入時，先保留現有 `checkIsAdmin / checkIsEditor / checkIsLeader` 作為 fallback，相容既有資料。
- 原因：現在專案已經大量使用這些 helper，完全抽掉風險較高。
- 替代方案：
  - 一次全面改成新權限 API：風險太高，回歸範圍過大。

### 5. 角色權限頁先做管理端能力，不先做員工端自助申請

- 決定：第一版只讓管理者設定角色與權限，不做申請流程或審批流程。
- 原因：目前需求是管理能力，不是流程系統。
- 替代方案：
  - 同時做權限申請：會把範圍拉太大。

## Risks / Trade-offs

- [權限相容期邏輯變複雜] → 先把舊 helper 包進新的 permission resolver，逐頁替換，避免同時改太多。
- [設定頁區塊很多，統一風格時容易漏掉小區塊] → 先從頂部工具列與主要管理區開始，留測試與 Chrome 驗證截圖。
- [通知全部已讀若只更新部分資料，badge 可能不同步] → 更新後統一從通知來源重新計算未讀數。
- [角色權限定義不完整，可能導致頁面看得到但按鈕不能用] → spec 內要明確區分 page access 與 action permission。
- [參考系統與本專案規模不同] → 只借結構，不直接照搬所有欄位與流程。

## Migration Plan

1. 定義角色權限資料結構與前端 resolver。
2. 先讓設定頁可以管理角色與使用者指派。
3. 將導航顯示接到新 resolver。
4. 將頁面內關鍵操作按鈕接到 action permission。
5. 補上通知中心「全部已讀」。
6. 最後整理設定頁風格與驗證。

如果上線後發生權限異常，可暫時切回舊 helper 判斷邏輯，因為這次保留相容層，不需要直接回滾全部 UI。

## Open Questions

- 權限資料要放在既有 `settings` 文件下，還是獨立 `permissions` 設定文件會更清楚？
- 使用者角色是否允許多角色同時存在，或先限制單一主角色？
- 設定頁中的哪一些操作一定要列入第一版 action permission？
