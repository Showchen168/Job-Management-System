## 1. 這次要做什麼

這次功能不是只做一頁，而是做一套完整流程：

1. 任務與問題都能留言
2. 全站右上角有通知鈴鐺
3. 員工可以從通知進到自己的卡片並處理留言
4. Leader 以上多一頁「團隊看板」，可看整組人底下有哪些卡片

一句話版本：
一般員工看自己的通知與自己的卡片，Leader 額外再看全隊。

## 3. 使用者怎麼用

### 3.1 一般員工

登入後會有這幾個主要入口：

1. 右上角通知鈴鐺
2. 待辦事項
3. 問題管理

一般員工不能進團隊看板，但一定要能：

- 在鈴鐺看到自己被 assign 的卡片
- 在鈴鐺看到哪張卡有新留言
- 點通知後直接進到對應卡片
- 在卡片內看留言、回留言、更新進度

### 3.2 Leader / Admin / Editor

除了上面一般員工有的功能，還多一頁：

- 團隊看板

這頁要能：

- 切換團隊
- 按人看卡片
- 一眼看出誰手上有什麼任務 / 問題
- 看哪些卡留言多、哪些卡剛被 assign

## 4. 權限規則

### 4.1 全員共用

- 通知鈴鐺：所有登入者都看得到
- 留言區：只要這張卡本來就在他權限內，就能看與留言

### 4.2 員工

- 只能看自己建立的卡，或指派給自己的卡
- 不能看團隊看板

### 4.3 Leader

- 能看自己所屬團隊成員相關卡片
- 能看團隊看板，但只看自己有權限的團隊

### 4.4 Admin / Editor

- 可看全部卡片
- 可看全部團隊

## 5. 畫面結構

## 5.1 全站共用頂部

要新增：

- 通知鈴鐺
- 未讀數字 badge
- 點開通知清單

通知清單每筆至少顯示：

- 類型：被 assign / 新留言
- 卡片標題
- 誰觸發
- 時間
- 是否未讀

## 5.2 一般員工頁面

沿用目前的 `待辦事項`、`問題管理` 頁，但補上：

- 頁首能看通知
- 卡片展開後出現留言區
- 點通知後能自動定位到那張卡
- 若是從通知點進來，該卡要自動展開

## 5.3 團隊看板

新增一個主選單頁：

- 名稱：`團隊看板`

畫面結構：

1. 團隊切換
2. 篩選器
3. 成員欄位
4. 每張卡片摘要

卡片摘要至少要有：

- 任務 / 問題類型
- 標題
- 狀態
- 到期日
- 留言數
- 最近動作摘要

## 6. 留言功能規格

## 6.1 留言顯示位置

留言不做獨立頁。

放在：

- 任務列展開區
- 問題列展開區

也就是使用者點開某列後，原本只有進度與描述，現在要多一塊留言區。

## 6.2 留言區內容

每則留言顯示：

- 留言者名稱
- 留言時間
- 留言內容

下方要有：

- 留言輸入框
- 送出按鈕

## 6.3 留言規則

- 自己留言不通知自己
- 留言送出後要更新：
  - 留言列表
  - 卡片上的留言數
  - 最後留言摘要
  - 對應收件人的通知

## 7. 通知鈴鐺規格

## 7.1 第一版通知來源

只做兩種：

1. `task_assigned`
2. `issue_assigned`
3. `task_comment`
4. `issue_comment`

## 7.2 未讀數規則

- 只計算目前使用者自己的未讀通知
- 使用者打開通知並進入卡片後，可標記該筆已讀
- 第二版再補「全部標記已讀」

## 7.3 點通知後的行為

點下通知後要做到：

1. 切到正確頁面
2. 找到對應卡片
3. 自動展開卡片
4. 若通知來源是留言，留言區要可直接看到
5. 該通知標記已讀

## 8. 團隊看板規格

## 8.1 顯示方式

這頁是「按人分欄」，不是按狀態分欄。

每個成員一欄，底下列出該成員的卡片。

## 8.2 篩選條件

第一版支援：

- 團隊
- 類型：全部 / 任務 / 問題
- 狀態

## 8.3 不做的東西

第一版先不做：

- 拖拉卡片
- 卡片欄位自訂排序
- 太細的進階篩選

## 9. 資料結構

## 9.1 留言

留言掛在卡片底下：

- 任務：`{task.path}/comments/{commentId}`
- 問題：`{issue.path}/comments/{commentId}`

欄位：

- `content`
- `createdAt`
- `createdByEmail`
- `createdByName`
- `updatedAt`
- `entityType`

## 9.2 卡片主文件摘要欄位

任務與問題主文件都補：

- `commentCount`
- `lastCommentAt`
- `lastCommentBy`
- `lastCommentPreview`

## 9.3 通知

建議存放位置：

- `artifacts/work-tracker-v1/public/data/users/{uid}/notifications/{notificationId}`

欄位：

- `type`
- `title`
- `message`
- `targetPath`
- `targetId`
- `targetType`
- `read`
- `createdAt`
- `actorEmail`
- `receiverEmail`

## 9.4 使用者摘要欄位

使用者文件補：

- `unreadNotificationCount`
- `lastNotificationAt`

## 11. 開發順序

### Phase 1

先做留言資料層與留言 UI

### Phase 2

再做通知鈴鐺與未讀邏輯

### Phase 3

最後做團隊看板

原因很簡單：
通知與看板都會吃到留言與卡片摘要資料，所以留言要先穩。

## 12. 測試順序

### 12.1 先補單元測試

- 留言摘要更新
- 通知建立
- 未讀數計算
- 團隊看板分組

### 12.2 再補元件測試

- 展開卡片看到留言區
- 送出留言成功
- 鈴鐺顯示未讀數
- 點通知能定位卡片
- 團隊切換後欄位資料正確

### 12.3 最後補 E2E

- 一般員工收到 assign 通知
- 一般員工收到留言通知
- 一般員工點通知進卡片回留言
- Leader 進團隊看板看成員卡片

## 13. 驗收標準

以下都成立，才算完成：

1. 任務與問題都能留言
2. 留言後卡片留言數會更新
3. 一般員工右上角看得到通知鈴鐺
4. 被 assign 時會產生站內通知
5. 有人留言時會產生站內通知
6. 點通知後會進到正確卡片並展開
7. 一般員工不能進團隊看板
8. Leader 以上可進團隊看板
9. 團隊看板能切換團隊並按人顯示卡片
10. 畫面風格維持目前專案的白底、圓角、藍色重點語言

## 14. 本次結論

這次不是做一個新頁，而是補齊一整條工作流：

- 鈴鐺提醒你
- 點進卡片
- 在卡片裡留言處理
- Leader 再用團隊看板看整體

這樣員工與 Leader 的動線才都完整，而且不會互相打架。
